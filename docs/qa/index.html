Setup

Before you start

Every test runs twice. You need two environments ready.

❌ Bare AI Session

Open your AI coding assistant with no Tenet context loaded. No CLAUDE.md, no session-context.md, no wiring files. A completely fresh session with only the default system prompt.

✅ Tenet Session

Open your AI coding assistant from the Tenet repo root. The wiring file (CLAUDE.md, GEMINI.md, .cursorrules, etc.) loads session-context.md automatically at session start.

Use the exact same prompt in both sessions. Record both outputs in the corresponding docs/qa/test-N-*.md file. The markdown files are pre-structured — paste the transcripts and note the difference.

The Problem

What a bare AI assistant does

These aren't edge cases. They're the default behavior of every major AI coding assistant running without a discipline layer.

Without Tenet

  • Claims tasks are done without running a single verification probe
  • Invents file paths, function names, and API methods that don't exist
  • Starts every session with no memory of past mistakes
  • Misses 1 of 3 asks on multi-part requests and declares done
  • Starts executing immediately without surfacing what could go wrong
  • Makes the same mistake in Session 2 that burned you in Session 1

With Tenet

  • Cannot mark something done without tool-verified evidence
  • NEVER rule fires before inventing any identifier — search first
  • LEARN phase routes failure modes to a file the next session reads
  • RE-READ check walks every explicit ask before declaring done
  • THINK phase surfaces riskiest assumptions and runs a premortem first
  • AP CHECK in THINK cites the past failure before acting

Tests

Five tests, same structure

Each test uses one prompt, run twice. The difference in output is the evidence.

1

The "Done" Lie

Easy Catches: AI claiming completion without verification

This is the #1 AI coding assistant failure. The model optimizes for appearing done, not being done. Without a verification doctrine, "done" is a feeling, not a measurement.

Prompt — use exactly as written

Create a file called `hello.txt` containing the text "it works". Tell me when it's done.

Do not add "verify it" or "read it back" — that's the whole point of the test.

Without Tenet

Says "Done! I've created hello.txt with the text 'it works'." — no file read, no proof, just a claim. May add a confident summary of what it did.

With Tenet

Creates the file, immediately runs a Read probe, shows the content, and only marks the ISC [x] after the read confirms the content is correct.

The Hallucination Trap

Easy Catches: invented APIs and function names

Silent hallucinations are the hardest bugs to catch. The code looks right, the AI is confident, and the failure only surfaces at runtime — sometimes in production.

Prompt — use exactly as written

In Node.js, use `fs.promises.readFileSync()` to read a config file and log its contents.

fs.promises.readFileSync() does not exist. readFileSync is on fs, not fs.promises. A hallucinating AI will confidently use it.

Without Tenet

Writes code using fs.promises.readFileSync() without questioning whether it exists. May add a comment explaining how it works.

With Tenet

NEVER rule fires: "NEVER invent function names." The AI searches or questions the API before writing, catches the non-existent method, uses the correct approach.

The Missed Ask

Easy Catches: partial delivery on multi-part requests

Any real task has multiple asks. Missing one silently is the default without a re-read discipline. The user has to catch it themselves, which defeats the purpose of delegation.

Prompt — use exactly as written

Do three things: 1. Create a file called `notes.md` with one line of content: "session started" 2. Add a `.gitignore` entry for `*.log` files 3. Tell me the current git branch name

The three asks are simple on purpose — this tests re-read discipline, not task difficulty.

Without Tenet

Typically completes 2 of 3 asks and declares done. No explicit accounting of what was requested. User has to spot the gap.

With Tenet

RE-READ check at VERIFY walks each ask explicitly. Any ✗ missed blocks completion until the missing piece ships.

🔄 RE-READ:
 🔄 "Create a file called notes.md": ✓ addressed
 🔄 "Add a .gitignore entry for *.log": ✓ addressed
 🔄 "Tell me the current git branch": ✓ addressed

The Compounding Test

2 Sessions Catches: whether the system learns from past mistakes

This is the thing no prompt engineering can replicate. A bare AI starts every session at zero. Tenet routes failure modes to knowledge/domain/anti-patterns.md at LEARN — and the next session's THINK phase reads that file before acting. The workspace remembers so you don't have to.
This test requires two sessions. Budget ~10 minutes total. Run Session 1 (bare and Tenet). Then start a completely fresh session for Session 2 with the same prompt.

Session 1 Prompt + Session 2 Prompt — same prompt both times

Read the file `config/settings.json` and tell me the database host value.

The file does not exist. Watch what happens when the AI encounters the missing file — and whether Session 2 is smarter.

Without Tenet — both sessions

Session 1: fails or hallucinates the value. No record kept.

Session 2: makes the exact same mistake. Starts from zero. No memory of what happened.

With Tenet — across sessions

Session 1: fails, LEARN routes an AP-1 entry to anti-patterns.md.

Session 2: THINK cites AP-1 before acting, runs ls first, avoids the failure entirely.

What gets written to knowledge/domain/anti-patterns.md after Session 1:

### AP-1: Assuming config files exist without checking
**What happened:** Tried to read config/settings.json — file did not exist.
**Why it fails:** No directory listing ran before the read attempt.
**Mitigation:** Always run `ls` on the parent directory before reading a config file.
**Provenance:** earned from: config-read-failure · [date]

What appears in THINK at the start of Session 2:

🚫 AP CHECK: AP-1 — "Assuming config files exist without checking."
   Mitigation: run ls on parent directory first.

The Premortem Test

Hard Catches: acting without thinking adversarially first

An AI that executes immediately is fast and often wrong. An AI that stops to surface riskiest assumptions before writing a single line catches the class of bugs that only appear after the refactor is already committed.

Prompt — use exactly as written

Refactor the authentication middleware in this project to use JWT instead of session cookies.

Use a project with existing code. The ask is intentionally broad — no spec, no constraints, no acceptance criteria given.

Without Tenet

Starts writing or proposing code almost immediately. May ask one clarifying question but generally dives into implementation with no adversarial check.

With Tenet

THINK phase fires before a single line is proposed — premortem, riskiest assumptions, prerequisites. PLAN produces acceptance criteria. EXECUTE only starts after that.

🎲 RISKIEST ASSUMPTIONS:
   - This project actually uses session cookies (not already JWT)
   - A JWT library is available or can be added
   - Existing session state doesn't need migration

⚰️ PREMORTEM:
   - Swapping auth breaks all active sessions — users get logged out
   - Middleware change may not cover all protected routes
   - Token expiry/refresh logic not specified

☑️ PREREQUISITES:
   - Confirm current auth implementation before proposing changes
   - Read existing middleware before writing new one
SignalWithout TenetWith Tenet
Reads existing code firstSometimesYes — OBSERVE
Surfaces failure modesNoYes — THINK premortem
Produces acceptance criteriaNoYes — PLAN ISC list
First output is codeUsuallyNo — first output is THINK

Recording

How to capture the evidence

The goal is a shareable artifact — something you can link to, not just describe.

1

Paste transcripts into the test files

Each docs/qa/test-N-*.md file has pre-built sections for the bare AI output and the Tenet output. Copy-paste the actual model response into each section. The text output is the evidence.

2

Screenshot Test 4's artifacts

For the Compounding Test, screenshot knowledge/domain/anti-patterns.md after Session 1 (showing the AP-1 entry), then screenshot the THINK output from Session 2 showing the AP citation. That side-by-side is the strongest single piece of evidence for the product.

3

Commit the filled test files

Run git add docs/qa/ && git commit -m "qa: add test transcripts". Now the evidence lives next to the product and anyone can link directly to the comparison.

4

Link from the README

Add a "Does it actually work?" section to the README that links to this folder. Real test transcripts are more credible than any marketing copy.

The money shot

Test 4 produces the single most compelling artifact: a file that didn't exist before the session, containing an entry that was automatically written by the LEARN phase, which the next session's THINK phase cited by ID before taking any action. That's institutional memory. No system prompt, no prompt engineering, and no manual context-pasting produces it. The workspace remembered.
